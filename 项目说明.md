## 一、这是什么

**graphrag-kg-agent**（展示名 **Archigraph 档图** = archive + graph，意为「把知识像档案归档成图谱」）——面向个人文档集合（技术论文 / GitHub 仓库文档 / 产品需求文档）的端到端 **Agentic GraphRAG** 系统：

```
上传文档 → 解析切块 → 向量化 → LLM 抽实体关系 → Neo4j 知识图谱
    → ReAct 检索-反思 Agent（向量召回 + 图谱邻域扩展，多轮自主决策）
    → 带可追溯引用的回答（每个 [n] 角标都能点回原始 chunk）
    → 前端像素 Agent 房间实时反映后端每一步在干什么
```

与普通 RAG 的两个差异化：

1. **Graph + Agentic**：检索不只有向量召回，还沿知识图谱做实体关系邻域扩展；且检索由 LLM 通过 function calling 自主决策（查什么、证据够不够、要不要换查询再查），不是固定单次流水线。
2. **引用可追溯是硬要求**：每个实体/关系带 evidence chunk，每个回答段落带 `[n]` 角标，角标必须落到真实原文片段——前端点角标即可看到来源 chunk 与位置。

**定位**：个人学习 + 简历展示 + 公开可复现。**仓库**：https://github.com/JS7B/graphrag-kg-agent （公开）

**技术栈**：Python 3.11+ / FastAPI / Pydantic · Neo4j 5.26 + Vector Index（Docker，不另起向量库）· OpenAI-compatible LLM（chat / embed / rerank，配置全走 `.env` 不绑厂商）· **自研轻量 ReAct 循环 + 原生 function calling**（不引 LangGraph / LangChain）· PyMuPDF · React 19 + Vite + TypeScript + Cytoscape.js。

│  routers/ 协议层（HTTP→Run，立即返 runId）                                │
│  runs/    编排层（Run 状态机 + 内存事件队列 + SSE fan-out）                │
│  执行层：  parsing/ → graph/ → extraction/（入库链路）                     │
│           qa/（Agentic RAG 问答链路） conversations/（多轮记忆）           │
│  clients/ 薄封装：llm.py（chat/chat_with_tools/embed/rerank）· graph.py   │
└──────────────┬───────────────────────────────────┬──────────────────────┘
          Bolt 协议                       OpenAI-compatible API
┌──────────────┴──────────────┐      ┌─────────────┴─────────────┐
│ Neo4j 5（Docker Compose）    │      │ LLM 服务（.env 配置）       │
│ 图谱节点/关系 + 2 个向量索引  │      │ CHAT / EMBEDDING / RERANK  │

后端所有异步任务走统一的「协议 → 编排 → 执行」三层：routers 把请求转成 Run 立即返回 `runId` 不阻塞；runs/tasks 用 `asyncio.to_thread` 包裹阻塞调用（保证 SSE 实时推送），全程吞异常并 emit error 终态；执行层纯业务逻辑不依赖 RunStore，靠注入的 `on_event` 回调上报进度，保持可单测。

## 三、图谱数据模型（Neo4j）

一张图同时承载**文档、知识、对话**三类数据，全部节点带唯一约束、幂等 MERGE 写入：

```
(:Document)-[:HAS_CHUNK]->(:Chunk)                     文档与切块
(:Chunk)-[:MENTIONS]->(:Entity)                        chunk 提到实体
(:Entity)-[:RELATES {type, evidence_chunk_id,          实体间业务关系
                     confidence}]->(:Entity)             （类型作属性，不拆关系标签）
(:Conversation)-[:HAS_MESSAGE]->(:Message)             多轮对话记忆
```

| 节点 | 关键属性 | 说明 |
|---|---|---|
| `Document` | document_id、doc_type、parse_status、chunk_count | 文档元信息与处理状态 |
| `Chunk` | chunk_id（`文档id#序号`，确定性）、text、**char_start/char_end/page/heading_path**、embedding | 位置字段是引用落地的凭证 |
| `Entity` | entity_id、name、entity_type、normalized_name、mention_count | LLM 抽取后归并去重 |
| `Conversation` | conversation_id、title、message_count | 会话（标题取首问截断 20 字，可手动重命名） |
| `Message` | message_id（`会话id#turn_index`，确定性幂等）、role、text、citations、embedding | 问答双方消息都向量化存图 |

**两个向量索引**（cosine，维度 = `EMBEDDING_DIM`）：`chunk_embedding`（`Chunk.embedding`，问答召回）与 `message_embedding`（`Message.embedding`，为未来历史召回铺路）。启动时 lifespan 幂等建约束/索引，并做**维度兜底**——索引维度与配置不符（如测试残留）则 DROP+重建，根治维度污染（lessons L6）。

## 四、入库链路（文档 → 图谱）

`POST /api/documents` 上传后异步执行，SSE 实时推进度（UPLOADING→PARSING→EXTRACTING→INDEXING→IDLE）：

```
runs/tasks.py::run_ingest
  ├─ parsing/   解析（md / txt / PDF(PyMuPDF) / 仓库目录）→ 切块
  │             MAX_CHARS=800、OVERLAP=150，每块保留 char_start/char_end/page/heading_path
  ├─ graph/     每块 embed（向量化文本 = 文档标题 + heading_path + 原文，text 属性仍存原文）
  │             → Document + Chunk 节点 + 向量写入 Neo4j（MERGE 幂等）
  └─ extraction/ 逐 chunk LLM 抽实体关系（JSON 模式，失败指数退避重试 3 次）
                → merge 归并（同实体多称谓合并、去重）
                → writer 写图谱（:Entity / :MENTIONS / :RELATES，关系带 evidence_chunk_id）
```

设计要点：chunk_id / entity_id 均为确定性 id，重复入库同一文档 MERGE 命中旧节点不翻倍；业务关系统一落 `:RELATES` 关系类型作属性（避免过早拆类型堆 schema）；扫描版 OCR 不在核心范围，解析层留扩展口。

## 五、问答链路（Agentic RAG，工程核心）

`POST /api/chat` 同样异步：路由建 Run 立即返回 `{runId, conversationId}`，后台跑 `qa/agent.py::answer_question_agentic`——**纯自研 ReAct 检索-反思循环**（默认 `max_turns=4`）：

```
messages = [system(ReAct 指令), *history(同会话近期对话), user(question)]
evidence_pool = {}                      # chunk_id → ChunkHit，跨轮去重累积
for turn in range(max_turns):           # for + 硬上限，防死循环
    msg = llm.chat_with_tools(messages, tools, tool_choice="auto")
    ├─ 无 tool_calls → 模型认为证据够了 → 生成最终答案（终止路径①）
    └─ 有 tool_calls → 逐个执行工具、结果回传 messages，证据进池，继续下一轮
# 达 max_turns → 用已有证据强制生成兜底答案（终止路径②）
```

**两个检索工具**（OpenAI function calling JSON Schema，模型每轮自主决策调哪个、传什么、何时停）：

| 工具 | 做什么 | 何时用 |
|---|---|---|
| `vector_search(query)` | embed → Neo4j 向量召回 top_k=10 → rerank 截前 5 | 查具体事实 / 技术细节 |
| `expand_entity(chunk_ids)` | 沿 `MENTIONS` 找实体，`RELATES` 扩 1 跳邻域 | 已有片段，挖实体关联 |

**引用可追溯闭环**（三段）：`build_context` 用 `enumerate(chunks, start=1)` 让正文 `[n]` 角标与 `Citation(index=n)` 共用同一序号生成 → LLM 在答案里引用 `[n]` → 生成后**输出净化**：正则提取正文真实出现的角标，只保留对应 Citation（LLM 编造的引用进不了结果）。每条 Citation 带 chunk_id / document_id / 位置 / 片段，前端点角标经 `GET /api/chunks/{id}` 回原文。

**健壮性设计**：
- **降级**：端点不支持 function calling（BadRequestError）→ 整体回退 `pipeline.py` 线性 RAG（固定 SEARCHING→CHECKING→WRITING）；**单次工具失败不降级**，错误字符串回传给模型自行决策（ReAct 哲学）。
- **rerank 失败静默降级**为按向量分截断，不影响主链路。
- **并发限流**：模块级 `asyncio.Semaphore(3)`，防多请求打爆 LLM rate limit。
- **跨线程安全**：agent 跑在工作线程（`asyncio.to_thread`），emit 用 `call_soon_threadsafe` 投回事件循环线程，保证 RunStore 单线程访问。

## 六、多轮对话记忆（对话进图谱）

同一会话内可连续追问（「它」「那个」等指代能被理解），历史持久化在 Neo4j、重启/刷新不丢：

```
POST /api/chat {question, conversationId}
  ├─ conversationId 为空 → 路由同步建会话（标题=首问截断 20 字），立即返 id
  ├─ 为空会话的缺省标题（"新会话"）→ 首问时自动用问题改名（手动改过名不覆盖）
  └─ run_chat：读近期历史（窗口=最近 6 条消息）→ 注入 ReAct messages
       → 答完写回本轮 user/agent 两条消息（各 embed 一次，幂等 MERGE）
```

关键约束与取舍：
- **注入窗口控 token**：只注入最近 6 条（约 3 组问答），全量消息仍存图谱不删；工具暂不主动召回历史（消息向量索引已铺路，留作增强）。
- **守引用红线**：system prompt 强约束「回答仍须基于本次检索到的文档片段并标 `[n]` 角标，不得仅凭历史记忆作答」——历史只作指代理解，不作答案来源。
- **幂等**：`message_id = 会话id#turn_index` 确定性生成，重试不翻倍。
- 会话 CRUD 接口见 §八；前端多会话侧边栏支持新建 / 切换（历史回灌复用现有渲染与角标逻辑）/ 行内重命名 / 删除（二次确认）。

## 七、RunEvent 事件流与前端实时可视化

**Stage 枚举（前后端锁定契约，12 个）**：`IDLE / UPLOADING / PARSING / EXTRACTING / LINKING / INDEXING / SEARCHING / CHECKING / WRITING / DELETING / REBUILDING / ERROR`。问答链路的 stage 由工具调用**动态映射**（vector_search→SEARCHING、expand_entity→LINKING、生成→WRITING），多轮检索会反复 SEARCHING/CHECKING——前端动画因此真实反映 Agent 的自主决策节奏。

**RunStore**：进程内 `dict[run_id → Run]` + 订阅队列，`append_event` 存历史、更新终态并向所有订阅者 fan-out；`subscribe` 先补投历史事件防漏；终态超 10 分钟 TTL 清理。Run 是瞬态进度信号不持久化，Document/Message 落 Neo4j 兜底。

**前端数据流（单向，红线）**：

```
后端 SSE ──RunEvent──> useRunEvents ──currentStage──> AgentRoom（像素小人）
                    └──events[]──> RunEventTimeline（运行轨迹）
```

`AgentRoom` 是前端招牌：深紫房间 + box-shadow 像素小人 + 5 个家具工位。第四版**行为队列状态机**——stage 映射为一段行为剧本（如 searching＝档案柜翻找→抱文件回桌→翻阅→循环），`useAgentPosition` 用 rAF 逐帧插值消费队列，**中断即转**（stage 快速连切时从真实当前位置丢队列换剧本，不跳变）；真实 idle 时加权随机做闲事（喝咖啡/踱步/瞌睡）。**硬红线**：stage 只能来自真实 RunEvent，前端绝不伪造状态；`prefers-reduced-motion` 时小人静立仅呼吸。演进过程与文件分工详见 `frontend/前端说明.md` §8。

其余视图：LibraryView（上传 → Run → SSE → 终态刷新；删除二次确认 + `?confirm=true`）、GraphView（Cytoscape.js 渲染 `GET /api/graph/*`，支持实体邻域展开与搜索）、SettingsView（`/health/deps` 连通状态）。无障碍经专项整改达 WCAG AA（焦点环 / 键盘可达 / reduced-motion 双层兜底等）。

| 接口 | 说明 |
|---|---|
| `GET /health` · `GET /health/deps` | 健康检查（Neo4j / LLM 连通状态） |

## 十、关键设计取舍

| 决策 | 理由 |
|---|---|
| 自研 ReAct 循环，不引 LangGraph/LangChain | 单 agent + 两个内部工具的场景，手写循环代码量小、零新依赖、编排 100% 自控；框架价值在多 agent/状态机/checkpointer，本项目用不上 |
| 向量存 Neo4j Vector Index，不另起向量库 | 图谱和向量同库，检索时「向量召回 → 图谱扩展」一次连接完成，零额外运维 |
| 业务关系统一 `:RELATES`，类型作属性 | 不为「图谱好看」堆关系类型；稳定后再判断是否拆分 |
| 会话记忆存图谱 + 注入窗口 | 复用 Neo4j 零新依赖；窗口（6 条）控 token，全量存图不删；不用向量召回历史（相关性不保证），简单可控 |
| 工具只做两个（搜索 + 扩展） | 决策空间小、可控、省 token；按需再加，避免过早堆砌 |
| 降级只在「端点不支持 tool calling」时 | 区分能力不支持（整体降级）与单次工具失败（回传给模型决策），不掩盖偶发问题 |
| RunStore 内存态 + TTL，不持久化 | Run 是瞬态进度信号，Document/Message 落 Neo4j 兜底；持久化的存储复杂度当前不值得 |
| 事件按工具类型动态映射 stage | 满足「动画状态来自真实 RunEvent」红线，前端真实反映 Agent 多轮决策 |
| 鉴权默认空跳过、部署填值启用 | 个人项目优先本地开发体验；DELETE 强制 confirm 防误删 |

更完整的后端取舍表见 `backend/后端说明.md` §9。

## 十一、快速开始

前提：Docker Desktop · Python 3.11+（conda）· Node.js 18+。

```bash
git clone https://github.com/JS7B/graphrag-kg-agent.git && cd graphrag-kg-agent
cp .env.example .env    # 必填：OPENAI_BASE_URL / OPENAI_API_KEY / CHAT_MODEL /
                        #       EMBEDDING_MODEL / EMBEDDING_DIM / NEO4J_PASSWORD
cd backend && pip install -r requirements.txt && cd ..
cd frontend && npm install && cd ..

docker compose up -d neo4j                                           # 1. Neo4j
cd backend && python -m uvicorn app.main:app --reload --port 8000    # 2. 后端
cd frontend && npm run dev                                           # 3. 前端 → localhost:5173
```

注意：`.env` 的 LLM 配置若仍是占位值，`/health/deps` 会显示 `llm: not_configured`，问答/抽取/入库全部跑不通；`CHAT_MODEL` 需支持 JSON 输出模式与 function calling（不支持后者会走线性降级）。完整步骤与常见问题见 [`运行说明.md`](运行说明.md)。

### 🔧 待优化（非阻塞，质量项）
- **实体召回率 57.1% < 70% 目标**：LLM 抽取漏部分技术名词（React/Vite/TypeScript 等），可调抽取 prompt 或加近义合并；其余指标达标。结合 `evals/` 单独迭代。
- **turn_index 并发隐患**：`_NEXT_TURN`（读 max+1）与 `_ADD_MESSAGE`（MERGE）非原子，同会话并发写理论上可能算出相同 turn。当前受 LLM 信号量(3) + 单用户串行写保护实际不撞，且幂等 MERGE 保证不翻倍。记录在案。
- **SSE 与 API Key 鉴权冲突（部署前待办，见 lessons L7）**：浏览器 `EventSource` 不支持自定义 header，部署配了 `API_KEY` 后 SSE 进度流会被 401 挡住；开发模式（API_KEY 为空）无影响。公网部署需另解（token 走 query / cookie 鉴权）。
- **会话标题**：现取首问截断 20 字 + 手动重命名，未来可让 LLM 生成更智能的标题。
- **测试向量索引隔离**：Neo4j「单 property 单向量索引」限制无法物理隔离，现用 teardown 恢复维度 + lifespan 启动维度校验双兜底（见 lessons L6 + backend/DEVLOG）。
