# 评估 DEVLOG

## 2026-07-16 建立人工复核质量夹具与图结构诊断
- 做了什么：新增不依赖 LLM 和 Neo4j 的人工复核夹具，计算实体精确率、关系语义精确率、来源完整率、复核覆盖率，并输出六类图结构诊断。
- 这是什么：precision 只在已经人工判断正确或错误的 accepted 候选上计算；结构诊断则把夹具当成一张小图，统计孤立点、叶子节点、低置信边、跨文档重名、组件规模和泛化 hub。
- 为什么需要：旧评估只知道关键实体有没有被召回、关系能不能成链，无法判断多抽出的实体是否真的错误，也无法解释图谱为什么显得杂乱。
- 为什么这么做：用显式正例、负例和未复核三态标签把“不在非穷举 ground truth 中”与“人工确认错误”分开；所有指标保留分子、分母和复核覆盖率，避免用一个漂亮百分比掩盖样本很小。
- 踩了什么坑：历史 `evals/report.md` 来自真实模型运行，不能拿手写夹具覆盖；因此新增独立 `quality_report.md`，并明确它只是确定性口径基线，不是生产质量快照。

## 2026-07-11 修正分句边界与实体召回口径
- 做了什么：把幻觉率使用的 Markdown 分句逻辑提取为纯函数并补充回归测试，同时让评估汇总分别输出 pooled 与 macro 实体召回率。
- 这是什么：Markdown 感知分句器只把真正的中英文句末标点和列表换行当成论断边界，会忽略围栏代码并保留引用角标；pooled 是把所有标注实体放进一个池子后计算总命中比例，macro 是先算每篇比例再等权平均。
- 为什么需要：原来的换行与标点正则会把 `Cytoscape.js`、`React 19.2`、URL 等拆成多个假论断，抬高无引用句数量；单一“实体召回率”标签也无法说明长文档是否获得了更高权重。
- 为什么这么做：分句器使用私有哨兵暂时保护点号和 URL 内部标点，并跳过代码围栏、去除列表前缀，使纯单元测试不依赖 LLM；汇总直接保存每篇精确命中数和标注数，pooled 继续承担硬阈值，macro 作为文档间均衡性的参考。
- 踩了什么坑：旧实现从已经计算出的浮点召回率反推命中数，四舍五入后可能产生统计漂移；改为保存精确整数计数后再汇总。后续审查还发现 URL 尾部角标会让保护范围越界、连续标点会产生纯标点论断，以及解析失败文档会被提前跳过并虚高召回率；对应修复都用纯回归测试固定边界。`evals/report.md` 继续保留为 2026-07-03 历史快照，避免在没有真实评估运行时伪造新结果。
